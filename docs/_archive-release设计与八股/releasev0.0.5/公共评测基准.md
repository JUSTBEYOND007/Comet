# 公共评测基准引入 + Verifier A/B 实验 后端设计

> V0.0.5 ①.5 的后端设计与面试讲点归档。只写后端,前端不涉及(评测体系无前端)。

## 一、问题与设计动机

### 真问题

V0.0.5 之前的 ① 评测体系 v1 是「自制集」—— 自建 22 道 RAG 题 + 17 段对话 + 30 个金标三元组。但简历讲点有硬伤:**「题是自己出的、跑出来 100% 缺说服力」**。

业界做 LLM/RAG 评测的标准做法:
- **C-MTEB**:中文 retrieval 业界标杆(MMTEB 中文子集),对照 bge-m3 / bge-large 等公认基线
- **HotpotQA distractor**:多跳问答标杆,验证「深度研究多跳子问题分解」;distractor 设置自带 8 个干扰段落,是 ② Verifier 的最佳测试场景
- **(原计划)LongMemEval**:EMNLP 2024 长对话记忆基准 —— **已下架,见下文**

### 评测三层结构(最终态)

```
L1 应用层(自制集,已完成)         证明:产品自洽、记忆评测以此为准(中文)
L2 中文检索基准(C-MTEB T2Retrieval) 证明:通用检索能力
L3 多跳推理(HotpotQA distractor)    证明:深度研究的多跳能力 + Verifier same/cross A/B 对照
```

> 原计划 L4 LongMemEval 下架。理由见「五、为什么砍 LongMemEval」。

│       ├── runner.py                  # 多跳 QA + 检索 Recall + verifier 接入
│       ├── qa_verifier.py             # LLM-as-judge(基于检索证据判 1/0,与 gold 独立)
│       └── README.md

每个 benchmark 子模块导出 `run_benchmark(...)` 异步函数,与 `tasks/*` 协议一致,便于 `reporters.py` 复用。

## 三、关键设计决策

### 1. 数据获取统一走 HuggingFace `datasets`

- `load_dataset(...)` 首次下载缓存到 `eval/cache/hf_datasets/`(`.gitignore`)
- 国内访问慢 → 文档明确推荐 `HF_ENDPOINT=https://hf-mirror.com` 镜像(已踩坑)
- 数据集结构变化兜底:cmteb-t2 从 `C-MTEB/T2Retrieval` 迁到 `mteb/T2Retrieval`,且子配置的 split 命名从「同名」改成统一 `dev`,loader 适配新结构

### 2. 指标库不重写

L2/L3 都复用 `metrics.py` 的 `recall_at_k / ndcg_at_k / mrr` 纯函数。HotpotQA EM/F1 走官方 `normalize_answer` + token P/R/F1 口径,不自造。

### 3. 不引入 RAGAS 等评估框架

保持轻量。LLM-as-judge 自己写 prompt + 复用 `core/memory/json_utils.parse_json_object` 健壮解析。

### 4. 报告格式统一

所有 benchmark 都用现有 `reporters.write_benchmark_report / write_benchmark_details` 输出 `report-{name}-{ts}.md` + `details-{...}.json`,便于横向对比。

### 5. 结果按类型分目录

`results/rag/`(检索/推理)+ `results/memory/`(记忆)子目录,便于归档与查找。

### 6. 诚实污染声明

L3 HotpotQA README 写明「使用 dev 集,主流 LLM 训练截止可能晚于 2018 年发布,**存在污染风险**,仅作系统设计对比,不作绝对水平断言」—— 这种诚实性面试加分项。

## 四、② Verifier A/B 实验设计(本版精华)

HotpotQA runner 加 `--verifier {none, same, cross}` 接入:

```bash
# 基线:无 verifier
uv run python -m eval.run_eval --benchmark hotpotqa --sample 200 --verifier none

# 同模型 self-critique(deepseek 自评 deepseek)
uv run python -m eval.run_eval --benchmark hotpotqa --sample 20 --verifier same

# 跨家族 critic(deepseek 作 answerer,glm-4-flash 作 critic)
uv run python -m eval.run_eval --benchmark hotpotqa --sample 20 --verifier cross
```

`qa_verifier.py` 实现 LLM-as-judge:
- 输入:question + retrieved passages(top-4) + answerer's pred answer
- 输出:`{"pass": bool, "reason": str}` 结构化 JSON
- 评分维度:答案是否被 passages 支持(基于证据,不基于 gold,与 EM 独立)

统计三个对照指标:
- **Verifier 判过率**(judge_pass_rate):verifier 认为答得对的占比
- **漏检率**(false_positive_rate):verifier 判过但实际 EM=0 的占比 —— 越低代表 verifier 越可信
- **EM 一致率**(consistency_with_em):verifier 判断与 EM 0/1 同方向的占比

### 实测数据(简历可直接用)

**cmteb-t2(300 corpus / 30 query)**:

| 配置 | nDCG@10 | Recall@10 | MRR@10 |
|------|---------|-----------|--------|
| 纯向量 | 0.9699 | 0.9675 | 0.9778 |
| 纯 BM25 | 0.881 | 0.8558 | 0.9667 |
| **混合(向量+BM25)** | **0.9818** | 0.9675 | **1.0** |
| 混合 + rerank | 0.9777 | 0.953 | 1.0 |

**HotpotQA(200 题基线)**:EM=62% F1=74.49% 检索 Recall@4=95.75%

**Verifier A/B(20 题/组)**:

| 配置 | EM | F1 | Verifier 判过率 | 漏检率 | EM 一致率 |
|------|-----|-----|---------------|--------|-----------|
| same(self-critique) | 0.60 | 0.7175 | **0.0%** | 0% | 40% |
| **cross(跨家族)** | 0.70 | 0.7952 | **95%** | 31.6% | **65%** |

**核心结论**:`same` 自评判过率 0% 几乎全部拒回,EM 一致率仅 40% —— **「不能自评」的硬数据**。`cross` 跨家族判过率 95%,EM 一致率提升到 65% —— **「优先 cross,fallback same」决策的实证基础**。

## 五、为什么砍 LongMemEval

原本计划接入 EMNLP 2024 的 LongMemEval-S 作为 L4 长对话记忆基准。代码完整写完后实测发现:

- 2 题 oracle 子集**准确率全 0**
- 根因:本项目记忆萃取流水线**中文优先**(`core/memory/prompts/*.jinja2` 全中文 + 13 类中文实体/谓词受控词表),英文输入会触发 LLM 做「英文 → 中文翻译 + 萃取」,引入翻译噪声 + 实体名失真
- LongMemEval 的英文人名/地名被意译成中文,后续召回又用英文 query → 召不回来

工程判断:

> **「不卖做不到的事」**。删 `eval/benchmarks/longmemeval/` 整目录 + `run_eval.py` 相关分支与 `--lme-*` 参数 + 旧报告。记忆评测以 L1 自制中文集为准(17 段对话 + 19 题召回,实测 Recall@5=0.82 / MRR=0.73 / 三元组实体级 F1=0.82)。

**这条决策本身就是面试加分项**:面试官问「为什么不用 LongMemEval?」可以回答「英文集与项目中文优先定位不匹配,实测会失真,工程上明确砍掉而不是硬上 + 数据难看」。

## 六、HTTP 接口

`①.5` 是评测脚本,无 HTTP 接口。入口是命令行:

```bash
uv run python -m eval.run_eval --benchmark cmteb-t2 --corpus-limit 300 --query-limit 30
uv run python -m eval.run_eval --benchmark hotpotqa --sample 200 --verifier none
uv run python -m eval.run_eval --benchmark hotpotqa --sample 20 --verifier cross
uv run python -m eval.run_eval --benchmark all          # 两套都跑
```

## 七、面试讲点

1. **三层评测体系**:L1 自制(中文场景自洽)+ L2 C-MTEB(中文检索业界基线)+ L3 HotpotQA(多跳推理 + Verifier A/B),互相印证
2. **跨模型 Verifier 实验**:在 HotpotQA 上证明「same self-critique 不可用」,推动 ② 设计走 cross 路线 —— **离线评测直接驱动生产决策**
3. **EM/F1 走官方口径**:不自造指标,业界可比
4. **诚实污染声明**:HotpotQA 2018 年发布,模型可能已经训了,声明在 README 不藏着
5. **明确砍 LongMemEval**:不卖做不到的事,工程判断而非疏漏
6. **数据集结构兜底**:C-MTEB 迁到 mteb/T2Retrieval,loader 适配新 split 结构 —— 维护意识

## 八、易踩坑(已踩过)

| 坑 | 现象 | 解决 |
|----|------|------|
| C-MTEB/T2Retrieval 旧路径只剩 default config | `ValueError: BuilderConfig 'corpus' not found. Available: ['default']` | 切到 `mteb/T2Retrieval` |
| mteb/T2Retrieval 新结构 split 都叫 dev | `ValueError: Unknown split "corpus". Should be one of ['dev']` | corpus / queries / default 三 subset,split 全是 `dev` |
| LongMemEval 仓库文件无扩展名 | `datasets.load_dataset` auto-discovery 失败 | 用 `huggingface_hub.hf_hub_download` 直接下载 + 手动 JSON 解析 |
| 国内 HF 慢/断 | 下 157MB corpus 卡 30 分钟 | 文档推荐 `HF_ENDPOINT=https://hf-mirror.com` 镜像 |
| LongMemEval 实测 2 题全 0 | 流水线英文输入产翻译噪声 | 下架(见上文「为什么砍」)|

V0.0.5 ①.5 让 Comet 评测体系从「自评 100% 缺说服力」升级为「**自制 + 公共基准 + 跨模型对照实验**」三层硬数据,Verifier A/B 实验数据(`same` 0% vs `cross` 95% 判过率)成为 ② 设计决策的实证基础。
