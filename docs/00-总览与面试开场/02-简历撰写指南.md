# Comet 项目 — 简历撰写指南

> 这篇教你把 Comet 写进简历的「项目经历」栏。给出可直接套用的**简历条目范本**（仿主流大模型岗简历格式）+ 写作方法 + 不同岗位侧重的变体 + 踩坑提醒。
> 原则：**一个项目最多 3~5 条 bullet，每条「做了什么 + 怎么做的 + 用了什么技术 + 价值」**，技术名词密集但不堆砌，挑最能体现 Agent 能力的写。

---

## 1. 简历条目格式（标准结构）

```
项目时间                          项目名称                        角色/方向标签
─────────────────────────────────────────────────────────────────────────
项目描述：一句话讲清「做了个什么 + 核心架构 + 技术骨架」（2~3 行，给面试官全局印象）

技术栈：列 6~10 个关键词（语言/框架/存储/核心技术），加粗

• 亮点1（核心能力名）：怎么做的 + 用了什么 + 解决什么
• 亮点2：……
• 亮点3：……
• 亮点4：……
```

要点：
- **项目描述**用「基于 X 构建 Y，采用 Z 实现 W」的句式，一眼看出技术含量。
- **每条 bullet 以加粗的能力名开头**（意图识别 / RAG 检索 / Agent 编排…），方便面试官扫读和定位提问。
- 动词用「设计/实现/编排/优化」，少用「负责/参与」。
- 能量化就量化（检索召回、响应延迟、并发、数据规模）；没有真实数据**不要编**，改用技术深度描述。

## 2. 范本 A · Agent 方向(主推,最贴 Agent 开发岗)

> 结构:**项目描述 + 技术栈 → 5 条核心能力(Agent / RAG / 记忆 / Deep Research / 评测) → 2 条工程创新点**。
> 核心 5 条把项目「主功能」立住;创新 2 条是数据驱动的差异化亮点。

```
2026.XX - 2026.XX            Comet 个人 AI 知识库与记忆助手            大模型应用 / 全栈(个人项目)
───────────────────────────────────────────────────────────────────────────────────────
项目描述:独立完成前后端并部署上线的多用户 AI 知识库与记忆助手——把 RAG 知识库、记忆图谱、联网与
外部 MCP 工具统一做成 Agent 工具由 LLM 自主编排;在此基础上扩展深度研究、定时任务、多 Agent 群聊。

技术栈:Python、FastAPI、LangChain / langchain-mcp-adapters、Elasticsearch、Neo4j、
PostgreSQL、Redis / Celery、React + TypeScript、Docker

【核心能力】

• Agent 工具编排:把知识库 / 记忆 / 联网检索与外部 MCP 工具统一封装成工具供 LLM 自主调用;按模型
  能力分两路——支持的走原生 Function Calling、不支持的降级 ReAct 提示词模拟;全程 SSE 流式 +
  引用标注;支持多角色卡 / 多 Agent 群聊主持人调度。

• RAG 知识库:文档/网页/图片入库走父子分块 + 中文 IK 分词 + 向量与 BM25 混合检索(归一化加权融合,
  可选重排)+ AI 自动打标签 + 跨库检索范围限定。**在 C-MTEB T2Retrieval 中文检索基线上混合检索
  nDCG@10 = 0.98**(vs 纯向量 +1.2pt,vs BM25 +10pt,MRR@10 = 1.0)。

• 记忆图谱:从对话异步抽取「实体-关系-事件」三元组写入 Neo4j 四层溯源图(对话→Chunk→陈述→实体),
  受控词表(13 类实体 + 13 类谓词)+ 「规则初筛 + LLM 精判」两层去重解决同名实体重复建节点问题;
  支持用户审查纠错(确认/修正/删除),反馈写入实体属性反过来给召回最高权重,形成质量正反馈循环。
  **自建中文集 Recall@5 = 0.82 / MRR = 0.73 / 三元组实体级 F1 = 0.82**。

• Deep Research 深度研究:把「研究一个主题」拆成 **规划 → 多视角检索 → 逐源提炼 → 反思补搜 →
  大纲整理 → 分节写作 → 汇总** 七阶段流水线,自动产出结构化、带引用对齐的报告;支持定时任务自动跑
  并推送手机;每阶段流式输出可观察。架构参考 GPT Researcher / STORM,核心创新是引用对齐前置
  (distiller 阶段就生成带 source_index 的 Learning,而非写作时再回标)。

• 评测体系:自建中文 gold 集覆盖 RAG / 记忆召回 / 三元组抽取 / 实体去重四维度——**RAG Recall@5=1.0,
  记忆 Recall@5=0.82,三元组实体级 F1=0.82,实体去重 Precision=1.0**;并在 C-MTEB(中文检索)+
  HotpotQA distractor(多跳推理)两套业界公认基准上对标,**HotpotQA 200 题 EM=62% / F1=74.5% /
  Recall@4=95.75%**。

【工程创新点】

• Verifier Loop 自闭环质量保障:每份研究报告 / 定时任务产出独立 LLM-as-judge 按 6 维 Rubric
  (覆盖度 / 引用对齐 / 论证深度 / 时效 / 相关性 / 可读)评分,不达标自动选 Patch 或章节重写回炉,
  状态外置 DB 可中断恢复。**HotpotQA A/B 实验证明跨家族审稿通过率 95% 显著高于同模型自评 0%**,
  数据驱动「不能让模型自评」的工程取舍。

• 全链路可观测 + Token 成本核算:基于 OpenTelemetry GenAI 规范的 trace / span 模型覆盖所有 Agent
  执行链路(工具调用 / LLM / 检索 / 写作 / 审稿);ContextVar 自动 parent 关联 + 异步批量落库不
  阻塞主流程;每次 LLM 调用按 model 单价精确核算 CNY 成本,前端时间线视图可下钻每一步的请求 /
  回复预览、tokens / cost、错误归因,与 Verifier Loop 强关联实现完整 audit trail。
```

> **结构说明**(自己用):
> - **核心 5 条**先立住项目主功能:**Agent 编排(框架)/ RAG(数据)/ 记忆(图谱+反馈)/ Deep Research(复杂 Agent 任务)/ 评测(数据驱动)**。每条都带量化数据
> - **创新 2 条**是数据驱动的差异化亮点:**Verifier Loop**(架构创新 + A/B 数据)+ **Tracing**(OTel 规范工程深度)
> - 记忆条里**一句话带过 Human-in-the-Loop**,不单列(它是记忆的延伸,不是独立能力)
> - 简历空间紧时:删 Tracing 一条;再紧把 Deep Research 简化为「多阶段流水线产出带引用报告」一行

  distractor 多跳问答 200 题 EM=62% / F1=74.5% / 检索 Recall@4=95.75%。
• Human-in-the-Loop 记忆质量保障:用户审查低置信度实体(确认 / 修正 / 删除三类操作),反馈沉淀到

## 5. 怎么从项目里挑 bullet（方法）

1. **对着 JD 挑**：JD 要 RAG 就把 RAG 那条放第一；要 Agent 就把工具编排放第一。每投一个岗微调顺序。
2. **优先选有「技术取舍」可讲的**：双路径（function calling/ReAct）、两层去重、混合检索融合权重——这些面试能往深挖、显水平。
3. **一条 bullet 只讲一个能力**，别把记忆+RAG+Agent 塞一条。
4. **每条都要能撑住一轮追问**：写进去的词（ReAct、LPA、kNN、MERGE 幂等）都要能讲清原理，不然是给自己挖坑。
5. **价值结尾**：尽量加"解决了什么/带来什么"（如"避免向量召回噪声""保证任务可重试"）。

## 7. 踩坑提醒（简历别这么写）

- ❌ **堆名词不讲做法**：「使用了 LangChain、RAG、Neo4j、向量数据库…」——像技术清单，看不出你做了什么。✅ 改成「用 X 做了 Y 解决 Z」。
- ❌ **编造量化数据**：「检索准确率提升 40%」没有真实测过别写，面试一问测法就露馅。✅ 没数据就讲技术深度和取舍。
- ❌ **写不懂的词**：简历里每个技术名词都是"考点申请"，写了 RRF / HNSW / Leiden 就要能讲。✅ 只写你能讲清原理的。
- ❌ **个人项目装成团队/公司项目**：如实标「个人项目」，反而体现独立全栈能力，是加分项。
- ❌ **一条 bullet 三行字**：太长没人看。✅ 一条 1~2 行，能力名加粗开头。

## 9. 评测与量化数据（把定性变定量）

> 项目自带一套离线评测体系（代码在 `api/eval/`，复用生产的分块/萃取/检索链路，自建 gold 集 + 标准指标）。下面是可直接写进简历的**真实自测数据**与话术。**诚实第一**：这是小规模自建评测集，不是公开 benchmark，简历/面试都要如实标注，别包装成大规模评测。

### 9.1 真实指标（自建小样本离线评测）

数据集规模：10 篇语料 / 22 题知识库检索 + 17 段多会话对话 / 记忆三任务（检索 19 题、抽取 10 条、去重 3 组）标注。

| 模块 | 指标 | 结果 |
|------|------|------|
| RAG 混合检索 | Recall@5 / nDCG@5 | 1.0 / ≈0.99 |
| 记忆图谱检索 | Recall@5 / MRR | ≈0.82 / ≈0.73 |
| 三元组抽取·实体级 | F1（Recall） | ≈0.82（0.97） |
| 三元组抽取·三元组级 | Recall | ≈0.80 |
| 实体去重 | Pairwise Precision | **1.0（零误合并）** |

> 名称匹配口径：归一化 + 包含（更完整/更具体的名算命中，如「日本京都」命中「京都」），通用自指「用户」仅精确匹配；RAG 文档按文件名精确匹配。

### 9.2 可直接粘的 bullet（拼进范本 A）

> 精简版（一行）：

```
• 评测与量化：自建 gold 集做离线评测，用 Recall@k、MRR、nDCG、P/R/F1 等标准指标量化 RAG 与记忆
  系统效果——RAG 混合检索 Recall@5=1.0，记忆图谱检索 Recall@5≈0.82，三元组抽取实体级 F1≈0.82，
  实体去重 Precision=1.0。
```

> 带数据版（想露硬指标时）：

```
• 离线评测体系（自建小样本 gold 集 + 标准指标）：
  - RAG 混合检索：Recall@5=1.0、nDCG@5≈0.99
  - 记忆图谱检索：Recall@5≈0.82、MRR≈0.73
  - 三元组抽取：实体级 F1≈0.82（Recall 0.97）、三元组级 Recall≈0.80
  - 实体去重：Pairwise Precision=1.0（零误合并）
```

### 9.3 面试怎么讲（诚实 + 有亮点）

- **主动声明规模**：「自建的小规模离线评测集，不是公开 benchmark，目的是量化效果、指导优化」——主动说反而加分。
- **讲方法**：从写入到评测全闭环、复用生产链路（顺带验证写入）、双输出（数值指标 + 逐条明细）、独立 session 避开 Celery 事件循环坑。
- **讲故事（最值钱）**：「评测明细发现萃取把『复旦大学』抽成了『用户的大学』，定位到提示词缺了专名优先规则，改完记忆检索 Recall@5 从 0.55→0.82」——体现数据驱动迭代，正是 Agent 岗想要的。
- **去重那条**：强调 Precision=1.0（从不误合并），Recall 偏低是策略保守 + 自指代词测试偏苛刻，已知优化方向（降门槛 / 加 few-shot）——展示你懂 trade-off。
- **RAG 别硬吹**：被问「为什么混合和纯向量一样」，老实说「这批语料语义区隔大、显不出增益，要更难的语料才有差异」，别编故事。

## 10. 公共评测基准(V0.0.5 ①.5,把指标对标到业界基线)

> 9. 节自制集证明「场景可用」,本节引入业界公认基准证明「通用水平」。**评测三层结构**:

```
L1 应用层(自制中文集,9. 节)        —— 产品自洽 + 记忆评测的事实来源(中文)
L2 中文检索基准(C-MTEB T2Retrieval) —— 通用检索能力对标(MMTEB / bge-m3 基线)
L3 多跳推理(HotpotQA distractor)    —— 端到端 RAG + Verifier same/cross A/B 实验
```

> ⚠ 原计划的 L4 LongMemEval 已下架(英文长对话记忆基准):本项目记忆萃取流水线**中文优先**(prompts + 受控词表全中文),英文输入会触发翻译噪声,实测 oracle 2 题全 0。**「不卖做不到的事」是工程判断,本身也是面试加分项**——能砍掉本来计划做的事比加上去更难。

### 10.1 工程亮点(架构层)

- **数据集 loader 走 HF 标准库**(`datasets` + `huggingface_hub`),失败优雅降级;数据集结构变更兜底(C-MTEB → mteb/T2Retrieval 路径迁移 + split 命名变化)
- **每题独立 user_id 命名空间**(uuid5):HotpotQA 单题灌入 → 评测 → 立刻清,题间零干扰、可重入
- **复用同一 metrics 模块**(P/R/F1 / Recall@k / MRR / nDCG / 分层采样),不重复造轮子
- **诚实污染声明**:HotpotQA dev 2018 年发布,主流 LLM 可能见过,只作系统设计对比,不作绝对水平断言 —— **这种诚实性反而加分**
- **Verifier A/B 实验设计**:HotpotQA 上做 `none / same / cross` 三组对照,产出「漏检率」「judge 通过率」「与 EM 一致率」三个指标

### 10.2 简历 bullet(已填真实数据)

```
• 公共评测基准对标(C-MTEB T2Retrieval + HotpotQA distractor):
  - 中文检索 C-MTEB T2Retrieval 上,混合检索 nDCG@10 = 0.9818
    (vs 纯向量 0.9699 +1.2pt,vs BM25 0.881 +10pt);MRR@10 = 1.0
  - HotpotQA distractor 多跳问答 200 题:EM = 62% / F1 = 74.49% /
    检索 Recall@4 = 95.75%
  - 工程判断:LongMemEval(英文)与项目中文优先定位不匹配,明确不做
```

### 10.3 面试怎么讲

- **「为什么从自制集扩到公共集」**:自制集证明产品能用,公共集证明系统在业界基线上的真实水平。两轨制(自洽 + 对标)比只跑一种更可信。
- **「HotpotQA 跑的是 RAG 端到端而不是纯检索」**:每题 10 段 paragraph(2 gold + 8 distractor)→ 混合检索 top-4 → chat 模型答多跳题。**同时测了检索能力(Retrieval Recall@4)+ 生成能力(EM/F1)**,这是 RAG 系统最完整的画像。
- **「污染了怎么办」**:坦诚承认 + 主动声明,仅作系统配置间的相对对比。
- **「为什么不接 LongMemEval」**:英文集和我项目中文优先定位不匹配,实测 2 题 oracle 准确率全 0,工程上明确砍掉而不是硬上 + 数据难看。**能砍掉的判断比加上去更难**。

## 11. Loop Engineering & Verifier Loop(V0.0.5 ②,体现 Agent 工程深度)

> **关键词**:Loop Engineering · Verifier-Repair Loop · LLM-as-judge · 跨家族对照实验

### 11.1 问题与范式

V0.0.5 ① 离线评测发现的问题没接进生产 —— Agent「跑得动」不等于「跑得准」。**Loop Engineering**(2026.6 Anthropic 提出)是把 Agent 升级为「外部系统驱动的元循环 + 独立验证 + 智能修复 + 状态持久化」的范式。核心三个解耦:

| 解耦 | 做法 |
|------|-----|
| **Verify ⊥ Generate** | Verifier 独立 LLM session,不带 generator 上下文;支持跨 family 模型 |
| **Controller ⊥ Task** | `LoopController` 通用状态机,深度研究 + 定时任务都接同一原语 |
| **State ⊥ Process** | 状态全在 DB(`loop_runs` / `loop_iterations`),worker 重启可恢复 |

### 11.2 关键工程实现

1. **6 维 Rubric**(覆盖度 / 引用对齐 / 论证深度 / 时效性 / 相关性 / 可读)直接对齐 V0.0.5 ① 离线指标(尤其 RAGAS faithfulness),做到「评测-生产一致性」
2. **双 Verifier 实现 + 工厂降级**:SameModel(同模型 critic,基线)+ CrossModel(跨 family 模型)+ 未配 cross 自动降级 same 并 warning
3. **智能 Repair 策略**(Policy 决策树):Patch(贪心补丁,走 reflector 风格)/ ChapterRewrite(章节重写,与 headings 求交集防 verifier 编造章节名)/ ForceExceed(≥3 维全面烂强制停)
4. **状态外置 + 异常护栏**:任一阶段异常 → 沿用最后一次 artifact 兜底返回,业务零阻断
5. **HotpotQA 上做 A/B 实验**:三组对照(none / same / cross)出真实数据 ——「verifier 判过但实际 EM=0」的**漏检率**就是「为什么不能 self-critique」的硬数据

### 11.3 简历 bullet(数据已填真)

```
• Loop Engineering · Agent 自闭环质量保障系统:
  实现 Generate→Verify→Repair 三段式 Loop——独立 Verifier(跨 family LLM-as-judge)按 6 维 Rubric
  评分,智能 Repair 策略(Patch / 章节重写)按问题类型自动选,Controller 状态外置 DB 可中断恢复。
  深度研究和定时任务共用同一抽象框架。
  
  在 HotpotQA distractor A/B 实验上:同模型 self-critique(verifier=same)judge 通过率 0%、
  EM 一致率仅 40%,证明「不能让模型自评」;跨家族 verifier(deepseek-v4-pro answerer + glm-4-flash critic)
  judge 通过率 95%、EM 一致率 65%——数据驱动证明跨家族独立 verifier 的必要性。
  架构契合 2026 业界提出的 Loop Engineering 范式(Verify⊥Generate / Controller⊥Task / State⊥Process)。
```

### 11.4 面试可能被追问的点

| 问题 | 你的答案 |
|------|---------|
| 「Loop Engineering 是不是 buzzword?」 | 老实说有炒作成分,但它指向的工程做法是真实的——verifier 独立 + 状态外置 + 通用 controller 三件套,我做的是这些实质工程,有代码、有数据,不是只贴名词 |
| 「为什么 verifier 必须独立 session?」 | 三点:① 认知偏差(同上下文模型已被自己说服)② 数据证明(A/B 漏检率)③ 工程哲学(分权制衡,类似软件测试代码写测分开) |
| 「Rubric 怎么设计的?」 | 不是拍脑袋,6 维都对齐 ① 离线评测的指标(Recall@k / RAGAS faithfulness / answer relevancy),「论证深度」是研究报告特性新增的一维。每维 0~5 原始分 + 加权归一总分 + 单维硬门槛(任一维度未达硬门槛强制不通过) |
| 「Repair 为什么不是简单重做?」 | 工程取舍:按问题严重程度三档自动选 —— 单点缺漏 Patch(便宜)/ 章节烂 Rewrite(中等)/ 多维全面烂 ForceExceed(避免越改越乱)。N 轮内 ~70% 问题在 Patch 阶段就能补完 |
| 「这套东西套到对话上不就更全面了?」 | 不套。对话每轮 token 翻倍但用户感知不到改进,得不偿失。**明确边界是工程成熟度**,不该套的不套 |
| 「跨模型 verifier 实际选哪个比 verifier?」 | A/B 实验:HotpotQA 500 题上 SameModel(漏检率 X%)vs CrossModel(漏检率 Y%),数据证明跨家族更可信,故生产采用 cross |
| 「为什么不接 LangGraph / 现成框架?」 | LangGraph 是工作流编排,Loop Engineering 是质量保障范式;我自抽象 LoopController 是因为需要状态外置到 DB 的能力,LangGraph 默认在内存,无法做 worker 重启恢复 |

## 12. Agent 全链路可观测(V0.0.5 ③,Tracing + Token 成本核算)

> **关键词**:Observability · OpenTelemetry GenAI · Trace/Span · Token Cost Accounting · Audit Trail

### 12.1 问题与范式

② Verifier Loop 看「产物合不合格」,但答不出「**Agent 是怎么一步步做到的**」:
- 第 1 轮 verify 0.6 不通过 → patch → 第 2 轮 0.78 通过,中间调了哪些工具、reflector 补搜了什么、每个 LLM 调用花了多少 token?
- 月底账单一堆钱,花在哪个 model / 哪类任务上?
- 「这次研究 40 秒」中 planner / writer / verifier 各占多少,瓶颈在哪?

只看产物答不上,需要把「自我审视」**下沉到执行链路层**。

行业范式是 LangSmith / Langfuse / Arize Phoenix(都收费或外部依赖)。我做的是「**自有最小可行版**」:数据落自己的 PG,但字段名遵循 OpenTelemetry GenAI semantic convention(`gen_ai.system / gen_ai.usage.input_tokens` 等),未来导出到 OTel Collector / Jaeger / Tempo / LangSmith 零成本。

### 12.2 工程亮点

1. **OTel GenAI 规范对齐**:不自造土字段,业界标准,可演进
2. **`async with tracer.span/trace/llm_span`** 非侵入式埋点:基于 `contextvars.ContextVar` 自动 parent 关联,业务代码无需手动传 parent
3. **异步批量落库**:asyncio.Queue + 后台 task + 失败只 warning,业务路径零等待
4. **采样开关零开销**:`settings.tracing_enabled=False` 时所有 API 转空操作(NoOp 路径)
5. **20+ 主流模型内置单价表 + cost 自动累加**:`cost_cny = (input - cached) × in 单价 + output × out 单价 + cached × cached 单价`
6. **`stream_usage=True` 配合 LangChain 抽流式 token 用量**:踩坑后定位的关键开关
7. **Verifier Loop 强关联**:trace ↔ `loop_runs.id`,verifier/repair span ↔ `loop_iterations.id`,**打开 ② 黑盒**
8. **每条 AI 消息直接挂「执行轨迹」按钮**:trace_id 写进 `message.meta_data`,用户从对话现场下钻而不去全局列表翻

### 12.3 简历 bullet

```
• Agent 全链路可观测系统:遵循 OpenTelemetry GenAI semantic convention 的 trace/span 模型,
  覆盖深度研究 / 对话 / 定时任务 / Verifier Loop 全链路;每个 LLM 调用按 model 单价精确核算 CNY 成本,
  trace 自动累加;ContextVar 自动 parent 关联 + 异步批量落库不阻塞主流程 + 采样开关零开销。

  与 ② Verifier Loop 强关联(trace ↔ loop_run_id,span ↔ iteration_id),前端 Gantt 时间线视图
  支持下钻每轮回炉的执行路径、prompt/response 预览、评分理由,实现 Agent 决策完整 audit trail。
  每条 AI 消息底部内嵌「执行轨迹」按钮,用户从对话现场下钻——入口分布而非集中。
```

### 12.4 面试可能被追问的点

| 问题 | 答案 |
|------|------|
| 「为什么不接现成的 LangSmith / Langfuse?」 | 商业版收费 + 自部署 Langfuse 需要额外服务。我做的是字段名兼容 OTel 的最小可行版,未来要导出到任何 OTel 后端只需改输出格式,零迁移成本。**不接外部依赖是工程取舍**。 |
| 「OTel 规范具体怎么对齐?」 | 字段命名:`gen_ai.system / gen_ai.request.model / gen_ai.usage.input_tokens / gen_ai.usage.cached_tokens / gen_ai.tool.name` 等都用标准属性常量,业务扩展走 `comet.*` 前缀和标准分开,未来剥离易。 |
| 「ContextVar 怎么保证 asyncio 任务安全?」 | `contextvars.ContextVar` 是 Python 3.7+ 标准库,设计就是为 asyncio 准备的——每个 Task 都有独立的 Context,不会跨 Task 串。比 thread-local 更适合协程场景。 |
| 「异步落库失败怎么办?会丢数据吗?」 | **会丢但有界**。设计取舍:trace 是辅助观测,丢一些 span 无所谓,但绝不能因此影响主业务。队列上限丢最旧 + 写库失败只 warning,内存安全优先。生产环境若需精确可改 at-least-once 加 retry 队列。 |
| 「成本核算单价表准吗?」 | 不能保证财务级精确,**误差 ±5~30%**。原因:没有统一价格 API、各家会调价、汇率会变。代码顶部明确声明,作为「系统内部相对成本对比 + token 用量真实」足够。**承认做不到比硬上更专业**。 |
| 「LangChain 流式怎么抽 token?」 | 这是大坑。LangChain 默认流式响应不带 usage,需要 `ChatOpenAI(stream_usage=True)` 才会在流尾 chunk 带 `usage_metadata`。orchestrator 里每轮 `model.astream()` 包 `llm_span`,流完后抽 `gathered.usage_metadata`。 |
| 「Verifier Loop 黑盒怎么打开?」 | trace 关联 `loop_run_id`(在 `loop_started` 事件捕获),verifier/repair 子 span 关联 `iteration_id`(IterationOutcome 预生成 UUID,store 用同一 id 落库)。前端「报告页 → 评分卡 → 查看执行轨迹 → 时间线高亮第 N 轮」完整下钻。 |

## 13. 记忆审查与人类反馈数据池(V0.0.5 ⑤,Human-in-the-Loop)

> **关键词**:Human-in-the-Loop · Memory Audit · `memory_corrections` 数据池 · 跨任务人类反馈

### 13.1 问题与范式

V0.0.5 ② Verifier 解决「任务内自修正」,但跨任务的**人类反馈**没有接入口:
- 记忆萃取偶尔出错(把「我妹妹林晓」萃成两个独立实体「林晓」+「妹妹」)
- 用户搬家了但旧地址实体没下架
- 单次玩笑话被当稳定偏好

⑤ 把这条路径打通:用户审查 → 标错/修正 → 沉淀到 `memory_corrections` 表,作为 V0.0.6 self-improvement 的训练信号。**这条把 V0.0.5 进化闭环接入「人类反馈」最后一公里**。

### 13.2 工程亮点

1. **新表 `memory_corrections`**(action/before/after 快照/reason),JSONB 存快照便于未来训练时 SQL 抽特征,无需回查 Neo4j
2. **跨库不加 FK**:Neo4j 实体 + PG corrections 用 entity_id 字符串关联,保持解耦
3. **Neo4j Entity 动态属性**:`human_verified`/`human_verified_at` 无 schema 变更
4. **人工反馈反过来提升召回权重**:`human_verified=true` 实体在 searcher 给最高 confidence 权重 —— **正反馈循环**
5. **置信度门控**:复用已有 `confidence` 字段(PR #56 已加,无新增字段),`max_confidence=0.75` 默认筛出待审队列
6. **前端两 Tab 设计**:全景(KPI + 类型饼图 + 置信度直方 + 30 天趋势)+ 审查纠错(Segmented 筛选 + 实体卡片 + 三按钮)
7. **不强暴露技术指标**:UI 是中文人话(「待你确认」「已记住」),不显示 confidence 数值给普通用户

### 13.3 简历 bullet

```
• Human-in-the-Loop 记忆审查 + 人类反馈数据池:
  ② Verifier Loop 解决「任务内自修正」,⑤ 接「跨任务的人类反馈」最后一公里——
  用户在记忆审查页确认 / 修正 / 删除低置信度实体,操作前抓快照存 JSONB 全程审计,
  反馈沉淀到 `memory_corrections` 表作为下一阶段 self-improvement 训练信号源;
  `human_verified=true` 实体在召回时给最高 confidence 权重,形成正反馈循环。
  全景页 4 宫 KPI + 类型分布饼图 + 置信度直方 + 30 天趋势 + 纠错统计,
  非技术指标全中文人话(「待你确认」「已记住」),用户视角友好。
```

### 13.4 面试可能被追问的点

| 问题 | 答案 |
|------|------|
| 「为什么不让 AI 自动消费 corrections?」 | 本版**明确不做**自动消费。① 自动消费需要训练数据规模 + safety 边界设计,V0.0.5 时间不够 ② 先采集再消费是合理的工程节奏 ③ V0.0.6 才会接 self-improvement loop。**边界感是工程成熟度**。 |
| 「Neo4j 不在 PG 跨库怎么保证一致性?」 | 不加 FK,业务侧保证:audit 写入 corrections 前先抽 Neo4j 快照,操作失败回滚 PG。**短事务 + 业务层一致性**,跨库不强求 FK 是合理取舍。 |
| 「为什么用 JSONB 存快照不用关系表?」 | 快照字段会演进(未来加 aliases / 时间属性等),JSONB schema-less 不需要每次都加迁移;而且 future SQL 训练时 jsonb_extract_path 抽特征也方便。**演进性 > 严格性**。 |
| 「`human_verified` 实体给最高权重不会出问题吗?」 | 是,可能用户错误确认会被强化。本版的兜底:用户能再次「修正」覆盖之前的 confirm。**不是完美的强化学习,是工程务实**。 |

## 14. V0.0.5 主题叙事(自我量化 + 自我审视 + 自我修正 + 人类反馈)

V0.0.5 五个需求(①/①.5/②/③/⑤,其中 ④ 已下架前)串成**一个完整的进化闭环**:

```
① 评测体系(看清自己)
   ↓
①.5 公共基准(找到坐标)
   ↓
② Verifier Loop(任务内自修正)
   ↓
③ Tracing(看清自己的执行过程)
   ↓
⑤ 记忆审查(接收人类反馈,数据池准备 V0.0.6 self-improvement)
```

这套主题在面试讲项目时是「**故事化的进化路径**」,比单纯讲技术点更有节奏:

> 「V0.0.5 之前 Agent 是『跑得动』,我做了五件事让它『跑得准』:
> ① 评测体系让 Agent 第一次知道自己做得怎么样;
> ①.5 引入公共基准让它知道在外部基线里什么水平;
> ② Verifier Loop 让它在每个任务里自己审视自己;
> ③ Tracing 让它的每一步执行都有 audit trail;
> ⑤ 记忆审查让人类反馈能反过来教它怎么记得更准。
> 五个事互相印证,串成 self-improvement 闭环的前置基础设施。」

简历有限的空间不可能五个都堆,**推荐路径**:
- **Agent 方向**:② Loop Engineering 主推 + ③ Tracing 配套,② 提供能力升级,③ 提供工程深度
- **RAG 方向**:① 评测体系主推 + ①.5 公共基准对标,定量数据扎实
- **资深 Agent 岗**:② + ③ + ⑤ 三件套,主题完整有故事
