# 评测框架与自制集(RAG + 记忆)— 设计与面试

> 给项目装一台体检仪——离线、可复现、用业界标准指标量化 RAG 与记忆两套核心系统的效果。让简历从「我做了 X 功能」变成「**我用 Recall@5 / MRR / nDCG / P/R/F1 量化了 X 功能的真实效果**」。
> 对应能力域:**评测体系**(自制集层)。代码:`api/eval/`(独立模块,与生产解耦)。

---

## 0. 能力定位(对应招聘要求)

- 对应 JD:**「LLM 评测」「RAG 评测」「Recall@k / MRR / nDCG」「数据驱动迭代」「LLM-as-judge / RAGAS」**。
- 角色:V0.0.5 主线第一大需求——评测是 Verifier Loop(②)的「问题诊断器」 + Tracing(③)的「真值参考」 + 记忆审查(⑤)的「质量基线」。**没有评测就没有 V0.0.5 整套进化叙事的起点**。

---

## 1. 解决什么问题

### 1.1 之前的盲区

V0.0.4 之前 Comet 主要靠**主观体验**判断好坏:「这次回答看着不错」「这条记忆好像没记对」。但:
- **无法量化**:简历只能写「我做了 RAG / 记忆图谱」,写不出「检索效果 Recall@5 = 多少」
- **无法迭代**:改了 prompt / 改了分块策略,有没有变好不知道,完全靠感觉
- **无法定位问题**:萃取出错了不知道是哪类问题(实体识别错? 谓词选错? 时间漏抽?)

### 1.2 方案

建独立评测模块 `api/eval/`,**与生产解耦**——独立模型配置、固定 `EVAL_USER_ID`、写入到评测全闭环、双输出(指标表 + 逐条明细)、可复现、随时跑随时清。

## 3. 核心设计

### 3.1 与生产解耦的「自包含」架构

| 维度 | 实现 |
|------|-----|
| 模型配置 | 独立 `.env.eval`(EVAL_LLM_API_KEY / EVAL_EMBEDDING_MODEL 等),不读用户的 model_configs 表 |
| 用户隔离 | 固定 `EVAL_USER_ID`(UUID 常量),写入用这个 user_id 标记,跑完独立清,不污染生产数据 |
| 客户端构建 | `eval_config.py` + `clients.py` 直接建 LLMClient,绕开 `core/llm/resolver.py` 的 model_configs 查询 |
| 写入到评测全闭环 | **复用生产的分块/萃取/写入链路**——评测同时验证写入流程,一举两得 |
| 输出 | 双输出:`report-*.md`(指标表 + KPI 卡)+ `details-*.json`(逐条命中/未命中明细),便于定位问题 |
| 数据清理 | `teardown.py` 按 EVAL_USER_ID 清 PG + ES + Neo4j,可重入 |

### 3.2 数据集设计(对齐业界基线)

#### RAG 评测(对齐 BEIR / HotpotQA 题型)

`fixtures/`:
- **10 篇百科语料**(marie-curie / radium / nobel-prize / penicillin / great-wall / mount-everest / giant-panda / photosynthesis / solar-system / yangtze-river)
- **`gold/retrieval.json` 22 题**(单跳 + 多跳混合)

#### 记忆评测(对齐 LoCoMo / LongMemEval 风格)

- **`dialogues.json` 17 段一致人设的多会话个人对话**
- **`gold/extraction.json` 10 条三元组**(严格用受控词表谓词,避免 LLM 自由发挥)
- **`gold/memory_retrieval.json` 19 题**记忆召回
- **`gold/dedup.json` 3 组共 14 个聚类**实体去重

### 3.3 指标体系(`metrics.py`)

| 指标 | 用途 | 实现 |
|-----|------|-----|
| Recall@k / Prec@k | 检索召回与精度 | 集合命中率 |
| MRR | 检索平均倒数排名 | sum(1/rank_i) / N |
| nDCG | 检索排序质量 | DCG / IDCG |
| 集合 P/R/F1 | 抽取实体级 / 去重正确性 | 标准公式 |
| Pairwise P/R/F1 | 去重的两两对比 | 把聚类拆成所有 (a,b) 二元对再算 |

### 3.4 名称匹配口径(关键!)

**归一化 + 包含**:更完整 / 更具体的名算命中(如「日本京都」命中「京都」),通用自指「用户」**仅精确匹配**(防止任何 entity 都包含「用户」给虚假命中)。

RAG 文档按**文件名精确匹配**(语料库太小,语义匹配会假阳)。

## 4. 真实指标(自建小样本离线评测)

数据集规模:10 篇语料 / 22 题知识库检索 + 17 段多会话对话 / 记忆三任务(检索 19 题、抽取 10 条、去重 3 组)标注。

| 模块 | 指标 | 结果 |
|------|------|------|
| RAG 混合检索 | Recall@5 / nDCG@5 | **1.0 / ≈0.99** |
| 记忆图谱检索 | Recall@5 / MRR | **≈0.82 / ≈0.73** |
| 三元组抽取·实体级 | F1(Recall) | **≈0.82(0.97)** |
| 三元组抽取·三元组级 | Recall | ≈0.80 |
| 实体去重 | Pairwise Precision | **1.0(零误合并)** |

---

## 5. 数据驱动迭代的真实故事(简历最值钱)

评测明细发现:萃取把**「复旦大学」**抽成了**「用户的大学」**——定位到提示词缺了**专名优先规则**,改完记忆检索 Recall@5 从 **0.55 → 0.82**。

**这是数据驱动迭代的典型样例**——评测发现问题、明细定位根因、改 prompt 验证效果。面试讲这个故事比讲指标本身更值钱。

## 7. 健壮性细节(写代码踩过的坑)

- **萃取模型字段容错**:LLM 返回 JSON 不一致(有时缺 `confidence`、有时多余字段),用 `json_repair` + Pydantic 兜底,不能因解析失败炸掉整轮评测
- **`run_eval.py` 加模型可用性自检**:跑前先 ping LLM / ES / Neo4j,失败友好报错而不是中途崩溃
- **`--reset` 清旧重跑**:重复跑同一份 fixture 时,先清 EVAL_USER_ID 再灌入,避免历史数据干扰
- **rerank 端点兼容**:必须用 **OpenAI / Jina 兼容**端点,百炼 DashScope 不兼容 OpenAI rerank API,会报 4xx
- **独立 session**:评测脚本不能复用 Celery 的 SQLAlchemy session(任务级事件循环坑),自己起独立 async session

## 8. 设计取舍

| 取舍 | 选择 | 原因 |
|------|-----|------|
| 用 RAGAS / DeepEval 这类现成框架 | 不用(本版) | 集成额外依赖、字段映射麻烦;自己实现 P/R/F1/MRR/nDCG 也就 200 行,可控性高。RAGAS 端到端留 backlog |
| 自动跑还是手动跑 | 手动 | 个人项目,评测是开发期工具,不需要 CI 自动化 |
| fixtures 多大规模 | 22 + 19 题(精挑细选)| 小而精,定位问题快;大量数据靠 ①.5 公共基准对标 |
| 评测结果暴露到前端 | 不做 | 开发者本地工具,普通用户不需要;前端只展示用户视角的「我的记忆全景」(⑤)|
| 与生产共享 user_id | 不共享(固定 EVAL_USER_ID) | 防污染生产数据,跑完独立清,可重入 |

## 9. 面试讲点

1. **写入到评测全闭环**:不是只测「检索」,而是「分块→写 ES→检索」全链路,顺带验证生产写入,一份代码两份收益。
2. **双输出**(指标 + 明细):指标看趋势,明细定位根因。少了任一份都没用——光看 Recall=0.82 不知道是哪 4 题没召回。
3. **数据驱动迭代真实样例**:「复旦大学」被抽成「用户的大学」→ 改专名优先规则 → Recall 0.55→0.82,讲故事比讲指标值钱。
4. **诚实声明小样本**:不包装成大规模评测;面试主动说「这是自建小样本 gold 集,目的是量化效果指导优化,大规模看 ①.5 公共基准对标」——主动声明反而加分。
5. **架构解耦**:`api/eval/` 与生产代码零依赖耦合(独立 user_id / 独立配置 / 独立 session),不污染、可单独发布、不进生产镜像(`.dockerignore` 排除)。
