## 一、账号体系与鉴权

### 设计

- 注册：用户名查重 → bcrypt 哈希密码 → 落库。
- 登录：校验密码 → 签发 access(30min) + refresh(7d) 双 token。
- 鉴权：业务接口加 `Depends(get_current_user)`，从 Bearer token 解析用户。
- 数据隔离：所有业务表带 `user_id`，查询强制过滤。

### 关键实现

```python
# JWT payload 带 type 区分 access / refresh
payload = {"sub": user_id, "type": "access", "iat": ..., "exp": ...}

# get_current_user 校验 type 必须是 access（refresh 不能直接访问业务）
if payload.get("type") != "access":
    raise BizError("认证令牌无效或已过期", code=1011, status_code=401)
```

### 八股

**Q：为什么用 access + refresh 双 token？**
access 短期（30min）用于访问业务接口，泄露了影响时间短；refresh 长期（7d）只用于换新 access，不参与业务请求，降低暴露面。access 过期前端用 refresh 静默续期，用户无感。

**Q：JWT 和 Session 的区别？为什么选 JWT？**
Session 状态存服务端（内存/Redis），每次请求查；JWT 无状态，签名自包含用户信息，服务端不存。JWT 适合前后端分离、易水平扩展（无需共享 session 存储）。代价是签发后无法主动失效（需配黑名单或短有效期 + refresh）。

**Q：密码为什么用 bcrypt 不用 MD5/SHA？**
bcrypt 是专门的密码哈希，自带盐（salt）防彩虹表，且**计算慢**（可调 cost），暴力破解成本高。MD5/SHA 是快速哈希，为校验完整性设计，不适合存密码——太快，易被暴力/字典攻击。

**Q：怎么做数据隔离防越权？**
所有业务表带 user_id 外键；所有查询在 repository 层强制带 `WHERE user_id = 当前用户`。即使前端传了别人的资源 id，也查不到不属于自己的数据。

## 二、模型配置与 API Key 加密

### 设计

不在 `.env` 写任何 LLM Key。用户在前端「模型配置」动态添加，按 type（chat/multimodal/embedding/rerank/websearch）和 provider 分类，Key 加密入库，按用户隔离。

### API Key 安全全链路

```
前端填明文 Key
  → encrypt_secret(Fernet 对称加密) → 存 model_configs.api_key_encrypted
  → 接口返回 mask_secret(仅露尾4位) ← 永不返回明文
  → 实际调用时 decrypt_secret 解密
```

加密用的 `FERNET_KEY` 放 `.env`，一经生成不可变（改了之前加密的 Key 都解不开）。`.env` 里只有 `FERNET_KEY`（加密用）和 `JWT_SECRET`（登录用）两个系统密钥，各家 LLM 的 Key 都走前端配置加密入库。

### 多 provider 适配

五个 provider（openai/qwen/doubao/deepseek/zhipu）都走 **OpenAI 兼容协议**，差异只在 base_url。所以不需要为每家写适配器，统一用 `ChatOpenAI(base_url, api_key, model)` 即可。联网搜索（千帆/tavily）才需单独适配。

### 八股

**Q：对称加密 vs 非对称加密，为什么 API Key 用对称（Fernet）？**
对称加密（AES 等）加解密用同一密钥，快；非对称（RSA）公钥加密私钥解密，慢，适合密钥交换/签名。API Key 是「自己存自己用」，不涉及多方密钥交换，用对称加密（Fernet 基于 AES-128-CBC + HMAC）即可，又快又够安全。

**Q：FERNET_KEY 丢了会怎样？为什么不能改？**
Fernet 加解密用同一个 key。改了或丢了，之前用旧 key 加密的所有 API Key 都解不开，用户得重新填。所以要妥善保存、不进 git、生产环境固定不变。

**Q：为什么不把 Key 放 .env / 环境变量？**
多用户场景每个用户用自己的 Key，放 .env 没法区分用户；且明文存配置不安全。走数据库 + 加密 + 用户隔离才合理。

## 三、Celery 异步任务（高频踩坑）

### 为什么要异步

文档解析、记忆萃取要多次调 LLM + 写多个存储，耗时长（几十秒）。同步会阻塞请求。改成 Celery 异步：接口落库 + 派发任务立即返回，worker 后台执行，完成回写状态，前端轮询。

### 多队列

```python
task_routes = {
    "app.tasks.parse.*": {"queue": "parse"},    # 文档/图片解析
    "app.tasks.image.*": {"queue": "parse"},
    "app.tasks.memory.*": {"queue": "memory"},  # 记忆萃取
    "app.tasks.beat.*": {"queue": "beat"},      # 定时任务
}
```

按域拆队列，便于分别限流/扩容。Celery beat 跑定时：每日 22:00 生成回顾、每日 3:00 全量社区聚类。

### 核心踩坑：事件循环 + 连接池

**问题**：项目是异步的（asyncio），但 Celery 任务是同步入口。任务里用 `asyncio.run()` 跑异步流程时，每个任务起一个新的事件循环。而全局的 DB 引擎/ES 客户端是单例，连接池绑定在第一次创建时的事件循环上 → 在新循环里用会报「event loop is closed / attached to different loop」。

**解法**：
```python
# 任务内用独立引擎 + NullPool（不缓存连接，用完即弃）
def create_task_engine():
    return create_async_engine(database_url, poolclass=NullPool, ...)

# 任务结束 dispose 引擎 + 重置 ES 客户端
async def _run(...):
    engine = create_task_engine()
    try:
        async with session_maker() as session:
            await _parse(...)
    finally:
        await engine.dispose()
        await elastic.close()  # 关本任务循环内创建的 ES 客户端
```

**Windows 踩坑**：worker 必须用 `--pool=solo`。Windows 上 Celery 默认的 prefork 池有权限问题（WinError 5）。solo 是单进程串行执行，任务内部用 asyncio 跑异步够用；Linux 生产可用 prefork + concurrency 提并发。

### 八股

**Q：Celery 的架构？broker / backend 是什么？**
Celery = 生产者（发任务）+ broker（消息队列，存待执行任务）+ worker（消费执行）+ result backend（存结果）。本项目 broker 和 backend 都用 Redis（不同 db）。生产者把任务序列化丢进 broker，worker 拉取执行，结果写 backend。

**Q：任务失败怎么办？幂等怎么保证？**
任务里 try/except 捕获，失败把业务状态置 failed 并存 error_msg，前端可触发重试。幂等：解析任务重试前先按 source_id 删旧 chunk；图谱写入用 MERGE by id，重复写只更新不重复建。

**Q：为什么不自己用 Redis 写调度器？**
Celery 是成熟方案，自带重试、定时（beat）、多队列、监控。自研调度器重复造轮子且易出 bug。技术选型优先用成熟组件。

**Q：定时任务怎么做的？**
Celery beat 进程按 crontab 周期把任务投进队列，worker 执行。本项目每日 22:00 生成每日回顾、每日 3:00 全量社区聚类兜底（增量聚类在萃取后实时触发）。

## 四、FastAPI 工程化

### 分层

`controller → service → repository → model/db` 严格单向。横切能力放 `core/`。好处：职责清晰、易测试、改动局部化。

### 统一响应 & 全局异常

```python
# 成功
return success(data, "可选消息")   # {code:0, message, data}
# 业务失败
raise BizError("中文提示", code=业务码, status_code=HTTP码)
# 全局异常处理器兜底未捕获异常 → 统一 {code, message, data} 格式
```

### 依赖注入

FastAPI 的 `Depends`：`get_session`（每请求一个 DB 会话，请求结束自动关）、`get_current_user`（解析 JWT）。依赖可嵌套、可复用、易 mock 测试。

### lifespan

应用启动时初始化 ES 索引 + Neo4j schema（幂等），关闭时释放四个存储的连接池。用 `@asynccontextmanager` 的 lifespan 管理。

### 八股

**Q：FastAPI 为什么快？async 的意义？**
FastAPI 基于 Starlette + asyncio，IO 密集型场景（调 LLM、查 DB、查 ES 都是网络 IO）用异步能在等 IO 时切去处理其他请求，单进程高并发。配合 uvicorn(ASGI)。注意：CPU 密集任务不该放异步主线程，要丢给 Celery。

**Q：依赖注入的好处？**
解耦、复用（鉴权/会话等通用逻辑写一次到处 Depends）、易测试（测试时可覆盖依赖注入 mock）。

**Q：pydantic 的作用？**
请求/响应数据校验 + 序列化。schema 定义类型，FastAPI 自动校验入参、生成 OpenAPI 文档。pydantic-settings 还用于读环境变量配置。

## 五、四存储连接池

| 存储 | 连接池要点 |
|------|-----------|
| PostgreSQL | pool_size + max_overflow；`pool_pre_ping` 取连接前 ping 剔除失效连接；`pool_recycle` 防被 DB 端超时断开；`statement_timeout` 防慢查询挂死 |
| Elasticsearch | max_connections + 重试 + 超时 |
| Neo4j | max_pool_size + 连接超时 |
| Redis | max_connections |

**八股 Q：连接池为什么重要？pool_pre_ping / pool_recycle 解决什么？**
连接池复用连接避免频繁建连开销。但长连接可能被 DB/网络中途断开变「死连接」，直接用会报错。`pool_pre_ping` 取连接前先发个测试查询，失效就丢弃重建；`pool_recycle` 定期主动回收超过 N 秒的连接，赶在 DB 端 timeout 之前。两者一起保证从池里拿到的连接是活的。

## 六、安全要点

- **密码**：bcrypt 哈希（自带盐 + 慢哈希）。
- **JWT**：HS256 签名，access/refresh 分离。
- **API Key**：Fernet 对称加密存储，返回掩码。
- **数据隔离**：全表 user_id 强制过滤。
- **SSRF 防护**：网页导入时校验目标 URL，防止用户传内网地址（如 169.254/127.0.0.1）让服务端代发请求。
- **SQL 注入**：用 SQLAlchemy ORM 参数化查询，不拼字符串。
- **Cypher 注入**：Neo4j 查询用参数传值（`$param`），标签/关系类型不参数化但内联固定值。

## 七、部署

开发期：四存储用 Docker 跑，应用（api/worker/beat/web）本地手动起，方便热重载调试。
生产期：docker-compose 全容器化（含 api/worker/beat/web）。

环境变量两套模板：根 `.env`（docker-compose 用，容器间用服务名互联）、`api/.env`（本地裸跑用 localhost）。

**八股 Q：开发和生产环境怎么隔离配置？**
配置全走环境变量（pydantic-settings 读 .env），不硬编码。不同环境用不同 .env，代码不变。敏感密钥（JWT_SECRET/FERNET_KEY）不进 git，只在部署环境配置。

## 八、综合追问（项目级）

**Q：这个项目最难的点 / 最有挑战的地方？**
①记忆萃取的质量与一致性：从自然语言稳定抽出结构化三元组、跨多次萃取做实体去重消歧（同名快路径 + 规则初筛 + LLM 精判）。②Agent 工具编排的双路径兼容（强/弱模型）。③Celery 异步任务里的事件循环与连接池管理踩坑。

**Q：如果数据量大了怎么优化？**
①ES/Neo4j 向量检索本身是 ANN，可扩；②知识图谱可视化做聚合/按社区分页加载，不全量渲染；③Celery 加 worker 横向扩容、按队列隔离；④PG 加索引、读写分离；⑤热点查询加 Redis 缓存。

**Q：怎么保证 LLM 输出的稳定性？**
①低温度（萃取/分类用 0.1~0.2）；②受控词表约束输出空间；③prompt 给充分示例 + 严格 JSON schema；④健壮的 JSON 解析容错；⑤失败降级不中断（单步失败跳过，整体流水线继续）。

**Q：项目的可扩展性体现在哪？**
①分层架构，加新功能按层加文件即可；②模型配置抽象，加新 provider 只要它兼容 OpenAI 协议就零改动；③存储抽象（StorageBackend 接口 + 本地/OSS 工厂），切换存储改配置不改代码；④Agent 工具是插件式，加新工具实现一个 StructuredTool 注册即可。
