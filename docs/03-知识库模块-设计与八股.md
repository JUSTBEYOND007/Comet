# 知识库 RAG 模块 — 设计与面试八股

> 结合 Comet 知识库模块的真实实现，讲解 RAG / 向量检索 / 混合检索 / 分块 / rerank / 中文分词 等面试高频点。
> 对应代码：`api/app/core/rag/`（parser/chunker/es_index/es_store/search/classifier/image_describe/web_crawler）+ `api/app/tasks/parse.py`

---

## 一、模块在做什么

把用户的文档、网页、图片转成可被语义检索的知识，问答时能精准召回相关片段并溯源。这就是 **RAG（Retrieval-Augmented Generation，检索增强生成）**：用检索到的私有知识增强 LLM 的回答，解决大模型「不知道你的私有数据」和「幻觉」问题。

---

## 二、入库流水线（写入侧）

以文档为例（`tasks/parse.py`，异步执行）：

```
上传文件 → 存储(本地/OSS) + PG 记录元数据(status=pending)
  → Celery 解析任务：
      1. 取文件
      2. 解析为纯文本（PDF/Word/MD/TXT/HTML）
      3. 父子分块
      4. 子块向量化（用户默认 embedding 模型）
      5. 写 ES（先删旧 chunk 保证重试幂等）
      6. AI 自动分类打标签
  → status=done
```

要点：
- **异步**：解析慢，走 Celery，接口立即返回，前端轮询状态。
- **状态机**：pending → parsing → done / failed，失败存 error_msg 可重试。
- **幂等**：重试前先按 source 删除旧 chunk，避免重复。

## 三、父子分块（Parent-Child Chunking）

```python
# core/rag/chunker.py
CHILD_CHUNK_TOKENS = 256    # 子块：小，用于精准向量召回
PARENT_CHUNK_TOKENS = 1024  # 父块：大，提供更多上下文
CHILD_OVERLAP_RATIO = 0.1   # 子块 10% 重叠，避免切断语义
```

**为什么父子分块**（面试高频）：

- 块切太大 → 向量召回不精准（一个向量代表太多内容，语义被稀释）；
- 块切太小 → 召回精准但上下文不足（LLM 拿到的信息太碎）。
- 父子分块两全：**用小的子块做向量召回保证精准**，命中后**返回它所属的大父块提供完整上下文**（small-to-big 策略）。

实现：先按中英文句子边界切，再合并到目标 token 数；子块带 10% 重叠防止关键信息被切断在边界。token 计数用 `tiktoken`。

## 四、ES 索引设计

```python
# core/rag/es_index.py  索引 comet_chunks
{
  "user_id": keyword,          # 多租户隔离
  "source_type": keyword,      # document | image
  "source_id": keyword,
  "chunk_type": keyword,       # child | parent | image_desc
  "parent_id": keyword,        # child 指向 parent
  "content": {                 # IK 中文分词
      "type": "text",
      "analyzer": "ik_max_word",       # 写入：细粒度
      "search_analyzer": "ik_smart"    # 查询：粗粒度
  },
  "tags": keyword,
  "vector": {                  # 向量字段
      "type": "dense_vector",
      "dims": 1024,
      "index": true,
      "similarity": "cosine"
  }
}
```

要点：
- **单索引 + user_id 过滤**：个人版数据量小，不必每用户一个索引。
- **向量维度固定 1024**：与 embedding 模型（智谱 embedding-3 / 通义 v3）一致，cosine 相似度。
- **文档、图片共用一个索引**：用 `chunk_type` 区分（child / parent / image_desc）。

## 五、混合检索（检索侧，核心）

`core/rag/search.py - hybrid_search`：

```
1. 向量召回：query 向量化 → ES kNN 检索（top recall_size）
2. BM25 召回：ES match 全文检索（top recall_size）
3. 归一化 + 加权融合：fused = 0.6*向量分 + 0.4*BM25分（各自 min-max 归一化）
4. 可选 rerank：配了 rerank 模型则对候选重排
5. 取 top_k，返回父块内容（small-to-big）
```

**为什么要混合检索**（面试高频）：

- **向量检索**擅长语义相似（「猫」能召回「喵星人」），但对精确关键词、专有名词、数字不敏感；
- **BM25** 擅长关键词精确匹配，但不懂语义。
- 两者互补，加权融合后召回质量更高。这是工业级 RAG 的标配。

**归一化为什么必要**：向量分和 BM25 分量纲不同（向量是 0~1 的相似度，BM25 是无上界的 TF-IDF 类分数），直接相加会被 BM25 主导，所以各自 min-max 归一化到 [0,1] 再加权。

**Rerank**：用专门的重排模型（cross-encoder）对初筛候选做精排。初筛（向量+BM25）追求召回率（recall），rerank 追求精确率（precision），是「粗排→精排」两阶段。本项目 rerank 可配置可选——配了就用，没配回退融合排序。

## 六、中文分词（IK）

**为什么必须装 IK**（面试可讲的踩坑点）：

ES 默认分词器对中文按**单字**切（「知识库」→「知」「识」「库」），BM25 检索中文效果极差。装 `analysis-ik` 插件后：
- 写入用 `ik_max_word`（细粒度，「中华人民共和国」切出所有可能词），提高召回；
- 查询用 `ik_smart`（粗粒度，切成最少的词），提高精度。

ES 镜像基于官方 + IK 插件自定义构建（`docker/es/Dockerfile`）。

## 七、全局搜索的「相关度门控」

普通 RAG 检索是**召回导向**（top_k 最近邻，宁可多召回）。但全局搜索是**精确导向**——用户搜「小狗」不该返回不相关的「案例分析.pdf」。

问题：向量 kNN 永远返回 top_k 最近邻，min-max 归一化抹掉了绝对相关度。

解法（`min_vector_score` 门控）：全局搜索改纯语义余弦门控——ES cosine knn 的 `_score = (1+cos)/2`，换算回 `cos = 2*score-1`，只保留 `cos ≥ 阈值`（0.45）的结果，按余弦降序。BM25 单字噪声不达标即丢弃。这样没有相关结果时宁可不展示。**只全局搜索启用门控，RAG Agent 检索保持召回导向。**

## 八、图片多模态 & 自动标签

- **图片**：上传后用多模态模型生成 描述/OCR/物体/场景，把描述向量化写 ES（chunk_type=image_desc），实现图片语义检索。
- **自动标签**：解析后用对话模型给内容分类，已有标签优先复用（防标签膨胀），最多 2 个。

## 九、面试问答（八股）

**Q1：什么是 RAG？解决什么问题？**
RAG = 检索增强生成。先从知识库检索与问题相关的内容，把它作为上下文喂给 LLM 生成答案。解决两个问题：①大模型不知道私有/最新数据；②减少幻觉（答案有据可依，可溯源引用）。

**Q2：向量检索的原理？**
把文本用 embedding 模型转成高维向量，语义相近的文本向量距离近。检索时把 query 也向量化，用近似最近邻（ANN，如 HNSW）算法找最近的 top_k 向量。相似度常用余弦相似度。

**Q3：为什么用混合检索而不是纯向量？**
向量懂语义但对精确关键词/专名/数字弱，BM25 懂关键词但不懂语义，两者互补。加权融合（本项目 0.6 向量 + 0.4 BM25）召回质量优于单一方式。

**Q4：分块策略怎么定的？块大小有什么影响？**
块太大向量召回不精准，太小上下文不足。本项目用父子分块：256 token 子块做召回（精准），命中后返回 1024 token 父块（上下文足）。子块 10% 重叠防止语义被边界切断。

**Q5：rerank 是什么？为什么需要？**
重排模型（cross-encoder）对初筛候选做精排。初筛（向量+BM25 是 bi-encoder，快但粗）追召回，rerank 把 query 和每个候选拼一起算精细相关度，追精确。两阶段「粗排→精排」兼顾速度和质量。

**Q6：embedding 维度为什么要固定？**
ES 向量字段和 Neo4j 向量索引创建时就要指定维度，必须和 embedding 模型输出维度严格一致，否则写入/检索报错。本项目固定 1024（智谱 embedding-3 传 dimensions=1024）。换模型要重建索引。

**Q7：怎么保证多租户数据隔离？**
所有 chunk 带 user_id，检索时强制加 user_id 过滤（term filter）。单索引 + 过滤的方式，个人版数据量小足够，省去每用户建索引的开销。

**Q8：召回的相关性怎么保证 / 不相关结果怎么处理？**
普通问答检索召回导向（取 top_k）。需要精确时（全局搜索）加绝对相关度门控：只保留余弦相似度 ≥ 阈值的，低于阈值的最近邻当噪声丢弃。

**Q9：文档解析失败/重试怎么处理？**
状态机 pending→parsing→done/failed，失败存 error_msg。重试前先按 source_id 删除已写入的旧 chunk，保证幂等不重复。

**Q10：HNSW / kNN 了解吗？**
HNSW（分层可导航小世界图）是主流的 ANN 算法，ES 的 dense_vector 索引底层用它。它用多层图结构加速最近邻搜索，牺牲极小精度换大幅提速，适合大规模向量检索。`num_candidates` 参数控制搜索时的候选池大小，越大越准但越慢。

## 十、常见追问（进阶）

**Q1：向量和 BM25 的分数怎么融合？为什么用 min-max 不用 RRF？**
本项目用 min-max 归一化后加权（0.6 向量 + 0.4 BM25）。另一主流方案是 RRF（Reciprocal Rank Fusion，按排名倒数融合，不看绝对分），RRF 对分数分布不敏感更鲁棒。min-max 实现简单直观、可调权重，个人场景够用；要更稳可换 RRF。

**Q2：父块和子块都存进 ES 了，检索时怎么只召回子块？**
索引里用 `chunk_type` 字段区分（child/parent/image_desc），检索时 filter `chunk_type=child`（文档）只对子块做向量+BM25 召回；父块不参与召回，只在命中子块后通过 `parent_id` 反查取出来提供上下文。

**Q3：embedding 模型换了维度变了怎么办？**
ES 向量字段维度创建时固定，换维度必须重建索引（删 comet_chunks 重建）+ 重新向量化所有内容。所以维度是个要慎重定的全局参数（本项目固定 1024）。生产换模型是有成本的迁移操作。

**Q4：长文档解析慢、向量化慢怎么优化？**
①解析异步化（已做，Celery）；②子块向量化批量调用（embed 一次传多条而非逐条）；③大文档可分片并行解析；④向量化加缓存（相同内容不重复算）。

**Q5：检索召回不准，怎么排查优化？**
①看是召回阶段没召回到（recall 问题，调大 recall_size、检查分块大小、embedding 质量）还是排序问题（加 rerank）；②中文检查 IK 分词是否生效；③query 改写/扩展（HyDE、多查询）；④调融合权重。

**Q6：什么是 small-to-big / parent-child retrieval？**
检索用小块（精准定位），返回大块（充足上下文）。小块向量语义聚焦、召回准；但小块内容少，直接喂 LLM 上下文不够，所以命中后返回它所在的大父块。是 LlamaIndex/LangChain 都内置的经典策略。

**Q7：图片是怎么变成可检索的？**
图片本身不能直接向量化做语义检索。本项目用多模态模型先生成图片的文字描述（描述/OCR/物体/场景），把**描述文本**向量化存 ES（chunk_type=image_desc），检索时用 query 向量匹配描述向量。本质是「以文搜图」。

**Q8：知识库和记忆都用了混合检索，有什么异同？**
同：都是向量召回 + 关键词/全文召回 + 加权融合（0.6/0.4）。异：知识库在 ES 检索文本块、用 IK 分词；记忆在 Neo4j 检索实体、用 cjk 分词，且召回后多一步「一跳邻居关系遍历」补充结构化上下文。

**Q9：rerank 模型和 embedding 模型有什么区别？**
embedding 是 bi-encoder：query 和 doc 分别编码成向量再算相似度，doc 可预先编码入库，检索快，但 query-doc 没交互、精度有限。rerank 是 cross-encoder：query 和 doc 拼一起输入模型算相关分，有充分交互、精度高，但不能预计算、只能对少量候选实时算。所以「bi-encoder 粗筛 + cross-encoder 精排」。

**Q10：怎么防止检索到的内容超出 LLM 上下文窗口？**
控制 top_k（本项目问答工具取 top 5）+ 控制父块大小（1024 token）。极端长时可对召回内容再做压缩/摘要。也是为什么不能无脑召回大量内容塞给模型。
