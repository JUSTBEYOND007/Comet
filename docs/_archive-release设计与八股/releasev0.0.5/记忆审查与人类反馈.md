# 记忆审查页(全景 + 人类反馈数据池)后端设计

> V0.0.5 ⑤ 的后端设计与面试讲点归档。只写后端,前端组件不写。

## 一、问题与设计动机

### 真问题

V0.0.5 之前 Comet 的「记忆系统」(Neo4j 四层溯源图谱 + 三元组萃取 + 两层去重)已有 `confidence` 置信度字段(PR #56 合入),但用户**没路径告诉系统「这条记得错了」**:

- 把「我妹妹林晓」萃成两个独立实体「林晓」+「妹妹」
- 把过期信息当现状(用户搬家了,但旧地址实体没下架)
- 把单次玩笑话当稳定偏好

V0.0.5 主题是「自我量化 + 自我审视 + 自我修正 + 人类反馈」。② Verifier Loop 解决「任务内自修正」,⑤ 补上「**跨任务的人类反馈**」—— Human-in-the-Loop。

### 设计目标

让用户能:
1. **全景**看到「AI 关于我记住了什么」(实体/关系/事件按类型分布、置信度分布、近期趋势、纠错历史)
2. **审查 + 纠错**低置信度实体(确认 / 修正 / 删除三类操作)
3. 所有操作沉淀到 `memory_corrections` 表,作为 **V0.0.6 self-improvement loop 的训练信号源**

明确不做(V0.0.5 范围):
- 自动消费 corrections 反馈训练(V0.0.6 的事)
- 让管理员审查全平台数据(个人项目无此需求)
- 暴露技术指标到 UI(对普通用户违和)

## 三、数据模型

### `memory_corrections` 表(新增)

```sql
memory_corrections(
  id, user_id,
  entity_id,                -- Neo4j 实体 id(无 FK,因为 Neo4j 不在 PG)
  action[confirm|correct|delete],
  before_snapshot jsonb,    -- 操作前的实体快照(name/type/description/aliases)
  after_snapshot jsonb,     -- 操作后的实体快照(correct/delete 才填)
  reason text,              -- 用户填写的修正原因(可选)
  source_dialogue_id,       -- 来源对话(可选,本版未接,留扩展)
  created_at
)
```

设计要点:
- **不加 Neo4j FK**:实体在 Neo4j,记录在 PG,跨库无法 FK;用 `entity_id` 字符串关联,业务侧保持一致性
- **快照存 JSONB**:before/after 完整结构化字段,便于未来 V0.0.6 训练时直接 SQL 抽特征,无需回查 Neo4j
- **action 三档**:对应用户三种操作意图,语义清晰
- **reason 可空**:不强制写,降低用户操作摩擦

### Neo4j Entity 节点扩展

不需要 schema 迁移,Neo4j 是动态属性:

```cypher
// HUMAN_VERIFY_ENTITY:用户「确认」时调用
MATCH (e:Entity {id: $entity_id, user_id: $user_id})
SET e.human_verified = true, e.human_verified_at = $now

// CORRECT_ENTITY:用户「修正」时调用
MATCH (e:Entity {id: $entity_id, user_id: $user_id})
SET e += $patch                                 // 更新 name / type / description
SET e.human_verified = true, e.human_verified_at = $now

// ENTITY_SNAPSHOT:操作前抽快照,supplies before_snapshot
MATCH (e:Entity {id: $entity_id, user_id: $user_id})
RETURN e.id, e.name, e.type, e.description, e.aliases, e.confidence, e.memory_layer
```

`human_verified=true` 的实体在 `searcher._rank_memory_hits` 里**最高 confidence 权重**(本版顺手做,但默认走主动召回的余弦门控,不强干预排序)。

## 四、服务层 5 个方法

### `review_overview(user_id, days=30)`

聚合 Neo4j + PG,产出全景数据:

```python
{
    "total_entities": int,           # 实体总数
    "long_term": int,                # 长期实体数
    "verified": int,                 # 已人工确认数
    "pending": int,                  # 置信度 < 0.75 且未确认数
    "total_relations": int,
    "days": int,
    "type_distribution": [           # 类型分布(供饼图)
        {"type": "人物", "count": 12},
        ...
    ],
    "confidence_buckets": [          # 置信度直方 5 桶(供直方图)
        {"range": "0~0.4", "count": 0},
        {"range": "0.4~0.6", "count": 3},
        ...
    ],
    "trend": [                       # 近 N 天每日新增(供趋势图)
        {"date": "2026-06-21", "count": 5},
        ...
    ],
    "correction_counts": {           # 用户纠错累计动作数(供 Tag 横排)
        "confirm": int, "correct": int, "delete": int
    },
}
```

### `list_review_entities(user_id, max_confidence, type_filter, include_verified, limit=100)`

筛选低置信度实体供审查:
- `max_confidence`:阈值过滤,默认 0.75
- `type_filter`:实体类型筛选
- `include_verified`:是否包含已确认的(默认 false,只看未审)
- 返回:实体 + top-3 关系 + 是否已 human_verified + memory_layer

### `confirm_entity(user_id, entity_id)` / `correct_entity_with_reason` / `delete_entity_with_reason`

三类操作的统一模式:

```python
async def confirm_entity(self, user_id, entity_id):
    # 1. 拿 before_snapshot
    snapshot = await self.graph_repo.entity_snapshot(user_id, entity_id)
    # 2. Neo4j 写入(确认 / 修正 / 软删除)
    await self.graph_repo.human_verify_entity(user_id, entity_id)
    # 3. 落 memory_corrections 表(原始 + 反馈)
    await self.correction_repo.record(
        user_id=user_id,
        entity_id=entity_id,
        action="confirm",
        before=snapshot,
        after=None,
        reason=None,
    )
```

`_serializable` 工具把 Neo4j 返回的 datetime 等转 JSON 友好类型,避免 JSONB 序列化失败。

## 五、HTTP 接口

```
GET    /api/memories/review/overview?days=30           全景数据
GET    /api/memories/review/entities                   实体审查列表
       ?max_confidence=0.75&type=人物&include_verified=false
POST   /api/memories/review/entity/{id}/confirm        确认
PATCH  /api/memories/review/entity/{id}/correct        修正(body: name/type/description)
DELETE /api/memories/review/entity/{id}                删除(body: reason)
```

路由顺序:`/review/overview` / `/review/entities` 静态路径在 `/{memory_id}` 之前注册,避免 UUID 路由吞掉。

## 六、面试讲点

1. **Human-in-the-Loop 接入**:② Verifier Loop 是「任务内自修正」,⑤ 是「跨任务的人类反馈」,二者互补
2. **数据池 vs 实时消费**:本版只**采集** corrections 不做自动消费(V0.0.6 做训练消费),边界清晰
3. **跨库一致性**:Neo4j 实体 + PG corrections 用 entity_id 字符串关联,**不加 FK 保持跨库解耦**,业务侧保证一致性
4. **快照式审计**:操作前抓 before_snapshot 存 JSONB,可回滚 + 可作训练特征 + 可面试时讲 audit trail 设计
5. **置信度门控**:复用已有 `confidence` 字段(PR #56 已加,无新增字段),`max_confidence=0.75` 默认筛出待审队列
6. **人工反馈反过来提升召回权重**:`human_verified=true` 给最高 confidence 权重,**形成正反馈循环**
7. **不强暴露技术指标**:UI 是中文人话(「待你确认」「已记住」),不显示 confidence 数值给普通用户

## 七、易踩坑(已踩过)

| 坑 | 现象 | 解决 |
|----|------|------|
| Neo4j datetime 不可 JSON 序列化 | 落库 before_snapshot 时报错 | `_serializable` 工具递归转 str |
| ENTITY_LIST_ALL Cypher 没 human_verified 字段 | 列表里看不到「已确认」标记 | 加 `coalesce(e.human_verified, false)` 到 RETURN |
| 静态路由被 UUID 路由吞 | `/review/overview` 命中 `/{memory_id}` 返回 404 | 静态路由必须先注册 |
| 置信度直方桶口径不一致 | 前端「待你确认」点击跳 Tab 2 但筛选阈值对不上 | 直方图按区间 0.4/0.6/0.75/0.85 分桶,「待你确认」固定走 0.75 |

V0.0.5 ⑤ 把「人类反馈」从「没接入口」变成「**数据池 + 实时反过来提升召回权重**」,为 V0.0.6 self-improvement loop 准备好训练信号源,主题叙事**「自我量化 + 自我审视 + 自我修正 + 人类反馈」**完整闭合。
